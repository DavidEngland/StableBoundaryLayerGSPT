        dX[1] = (1.0 / C_skin) * (R_net + H_sensible - G_ground)

                    end
                end
            end
        end
    end

    return nothing
end

## 1. Critical Performance Bottlenecks

### Loop-Throttling `hasproperty` Lookups

Inside the primary `@inbounds @simd` loop, you call `_effective_h_scale(p, U_face, V_face)`. Inside that function, there are five separate `hasproperty` checks executed on every single iteration.
Because `p` is a runtime parameter struct, `hasproperty` relies on reflection or dynamic runtime field lookups. This completely destroys type stability, guarantees that the compiler cannot inline the function cleanly, and **shatters the `@simd` vectorization window**.

* **The Fix:** Hoist all configuration flags and scalar parameter defaults out of the loop entirely, passing only raw scalars into the local calculations.

### SIMD-Killing `hypot`

The `hypot(x, y)` function is designed to prevent overflow and underflow at the extreme limits of floating-point precision ($10^{\pm 308}$). To achieve this safely, it includes complex internal branching logic.
Because these variables represent physical boundary layer scales (well within safe numerical bounds), `hypot` introduces unnecessary overhead and blocks vectorization.

* **The Fix:** Replace `hypot(a, b)` with a direct `sqrt(a^2 + b^2)`, which maps beautifully to hardware vector units.

### Autodiff Allocation Leak

When calculating Jacobians via `ForwardDiff.jl`, the element type `T` becomes a `Dual` number. Because your preallocated workspace `ws.Km` is likely a `Vector{Float64}`, the condition `eltype(ws.Km) === T` fails, forcing the fallback path to allocate a brand-new vector on **every single RHS evaluation**. This ruins the performance of implicit solvers (like `Rosenbrock23` or `TRBDF2`).

## 2. The Half-Cell Discretization Bug

In your bottom boundary condition setup, you calculate the surface gradients like this:

```julia
dU_dz_surf = (U[1] - 0.0) / dz
dth_dz_surf = (theta[1] - T_s) / dz

```

Assuming a standard grid layout, `theta[1]` sits at the center of the first vertical cell ($z = \Delta z / 2$), while `T_s` is located precisely at the surface ($z = 0$).

The distance between the first cell center and the ground is a **half-cell step**, $\Delta z / 2$, not a full $\Delta z$. By dividing by `dz`, the surface gradients and subsequent surface fluxes are underestimated by exactly **50%**.

    flux_H_surf = K_h_surf * dth_dz_surf # Downward-positive internal flux

    @inbounds begin
        flux_U_top1 = K_m_faces[1] * (U[2] - U[1]) / dz
        flux_V_top1 = K_m_faces[1] * (V[2] - V[1]) / dz
        flux_H_top1 = K_h_faces[1] * (theta[2] - theta[1]) / dz

        dU[1] = f * (V[1] - Vg) + (flux_U_top1 - flux_U_surf) / dz
        dV[1] = -f * (U[1] - Ug) + (flux_V_top1 - flux_V_surf) / dz
        dtheta[1] = (flux_H_top1 - flux_H_surf) / dz

        # 7. Top Boundary Conditions (Fixed Half-Cell Step)
        dz_top = z_faces[N+1] - z_centers[N] 
        flux_H_topN = if theta_top_bc === :dirichlet
            K_h_faces[N-1] * (theta_top_ref - theta[N]) / dz_top
        elseif theta_top_bc === :relaxation
            -lambda_top * (theta[N] - theta_top_ref)
        else
            0.0
        end

        dU[N] = f * (V[N] - Vg) + (0.0 - K_m_faces[N-1] * (U[N] - U[N-1]) / dz) / dz
        dV[N] = -f * (U[N] - Ug) + (0.0 - K_m_faces[N-1] * (V[N] - V[N-1]) / dz) / dz
        dtheta[N] = (flux_H_topN - K_h_faces[N-1] * (theta[N] - theta[N-1]) / dz) / dz

        # 8. Surface Energy Balance (Standard Meteorological Sign Convention)
        # R_net: Positive Downward (warms surface)
        # H_upward & G_downward: Positive Upward/Away from skin (cools surface)
        R_net = R_down - sigma_SB * (T_s^4)
        H_upward = rho_cp * K_h_surf * (T_s - theta[1]) / dz_surf 
        G_downward = lambda_s * (T_s - T_deep) / d_soil 

        dX[1] = (one(T) / C_skin) * (R_net - H_upward - G_downward)
    end

    return nothing
end

## Strategic Architecture Shift: Moving Past Explicit Stiffness

Your point regarding **explicit vertical diffusion** hits the nail on the head. For stable nocturnal boundary layers where $K_h$ drops by orders of magnitude over fractions of a meter, an explicit treatment forces the adaptive ODE solver to drop its timestep to sub-second levels just to maintain absolute stability.

Even with highly optimized RHS code, the solver stalls.

If we wrap this system in a standard **IMEX (Implicit-Explicit) Split ODE problem** structure or build a tailored fractional-step algorithm, we can decouple the fast, stiff vertical diffusion operator from the slower advective/coriolis forcing terms.

Since we are targeting unconditional stability via your custom tridiagonal Thomas algorithm solver, how do you want to handle the non-linear coupling of the diffusion coefficients? Should we evaluate the operators via a classic **Richtmyer-Morton / Crank-Nicolson** setup lagged at time-step $n$, or should we set up a low-cost, internal Picard iteration loop to converge $K_m(X^{n+1})$ and $K_h(X^{n+1})$ implicitly?
